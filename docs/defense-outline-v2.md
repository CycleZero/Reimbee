### 第 7 页 · Agent 两大设计模式（⭐ 核心创新）

**左栏 🧩 装饰器模式**：

```
RoleGuard → Interruptable → 业务逻辑
```

- **零侵入包装**：任意工具加上装饰器即获得鉴权和确认能力
- 应用工具：`submit_reimbursement` / `approve_reimbursement` / `cancel_reimbursement`
- 不可逆操作前暂停，等待用户确认

**右栏 🤖 子 Agent 嵌套模式**：

```
check_compliance 工具
  → 创建独立 compliance_reviewer Agent
  → 内部 ReAct 循环（最多 5 轮）
  → 调用 search_policy（RAG 向量检索政策）
  → 返回 JSON：{level: pass/warning/error, rule_id, suggestion}
```

- 主 Agent 只看到干净的 JSON 判断结果
- 实现**关注点分离**与 Agent 组合复用
- 合规引擎同时有**规则引擎**（关键词+阈值）和**子Agent RAG**双路径

> 🎤 50s — 装饰器 20s + 子Agent 30s，这是答辩中最能体现设计深度的页面

### 第 9 页 · 两阶段中断确认机制（⭐ 核心创新）

**6 步流程**：

```
🤖 LLM 调用 submit_reimb
  → 🛑 Interruptable 检测首次调用
    → ⏸️ Agent 暂停，SSE 推送 interrupted 事件
      → 👤 前端弹窗"确认提交报销单？"，用户点击确认/取消
        → ✅ POST /api/chat/approve {approved:true}
          → 🔄 HandleApprove 重放工具，ApprovalState 单次消费
            → ▶️ Runner.Run(resume=true) 继续 ReAct 循环
```

**对比**：

| ❌ 无确认机制 | ✅ 两阶段确认 |
|-------------|-------------|
| LLM 直接调用敏感操作 | 阶段一：首次调用 → 暂停 → SSE 推送 interrupted |
| 理解偏差直接导致误操作 | 阶段二：用户确认 → HandleApprove → 重放 → 执行 |
| 不可逆操作无保障 | ApprovalState.Consumed 单次有效，防重放 |

**关键数据结构**：
- `ApprovalState{Approved, Reason, Consumed}` — 确认状态
- `session.State["pending_tool"]` — 中断时保存的工具信息
- `session.State["approval"]` — 恢复时的审批结果

> 🎤 40s — 流程 20s + 对比 10s + 数据结构 10s

### 第 11 页 · SSE 流式卡片渲染

**左栏：7 事件 → 4 卡片映射**

| SSE 事件 | 卡片类型 | 行为 |
|----------|----------|------|
| `thinking` + `reasoning` | 🧠 **ThinkingCard** | 默认折叠，展开看推理过程；完成后显示"思考完成" |
| `tool_call` + `tool_result` | 🔧 **ToolCard** | 默认折叠，22种中文标签，🟡执行中→🟢完成；OCR 工具内含**可编辑修正表单** |
| `message` (delta) | 💬 **MessageCard** | 始终展开，Markdown 表格渲染，流式增量更新 |
| `interrupted` | ⚠️ **InterruptBar** | 黄色警告栏，确认/取消按钮 + 补充说明 |

**右栏：核心机制**

```
fetchEventSource (SSE)
  → chatStore (Zustand)
    → appendCard() / updateLastCard()
      → MessageList → 4 种卡片组件
```

- **同类型合并**：连续 reasoning 分片追加到同一 thinking 卡片
- **类型切换弹新**：LLM 从推理→工具调用→回复，自动弹新卡片
- **v5 纯卡片模式**：`cards[]` 是消息的唯一视觉来源

> 🎤 40s — 左边映射 20s + 右边机制 20s
