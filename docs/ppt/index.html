各位老师好，我们小组做的是一个基于大语言模型 Agent 的企业智能报销系统，叫做 Reimbee。先简单说一下这个项目是干什么的——在传统的企业报销流程里面，员工要手动填报销单、贴发票、找领导签字，整个过程耗时很长，合规退回率也高。我们做的事情，就是把这些步骤全部用 AI 对话的方式来替代。员工只需要在聊天窗口里跟 AI 说"帮我报销这张发票"，系统就会自动帮你做 OCR 识别、合规审核、预算检查，然后生成报销单、提交审批，最后审批人也是通过对话来完成审批的。整个流程从一个多星期压缩到几分钟。后端用的是 Go 语言，前端是 React，AI 引擎用的是 DeepSeek 大模型加上 RAG 知识库做合规审核，整个系统打成单个二进制部署，一条命令就能跑起来。接下来我会从业务需求、系统设计、核心技术实现和测试效果四个方面，完整地给大家介绍这个项目。

先讲为什么要做这个项目。传统企业报销有三个最大的痛点。第一个是效率低，大家可能都有体会，一次报销从填单、贴发票到各级签字，平均要花十五分钟，如果一个月报销量大，几个小时就搭进去了。第二个是合规退回率高，差不多百分之三十的报销因为政策理解偏差或者审核疏漏被退回来重新改，反复折腾。第三个问题是流程完全不可见，提交之后进度在哪、谁在审批完全不知道。我们的目标很明确：第一，用 AI 替代百分之八十的传统流程，让员工只需要在对话中说几句话就能完成整条链路；第二，把单次报销压缩到三分钟左右；第三，合规退回率降到百分之五以下；第四，流程全程可追踪。核心思路就是从现在的"人去适应系统"变成"AI 主动为人服务"。

这是我们的系统总体架构图。自下往上看：最底层是数据层，MySQL、Redis、MinIO，还有一个 pgvector 做向量存储；往上是基础设施层，我们做了大量可插拔的接口设计，OCR 支持千问 VL、Paddle 两种后端，向量嵌入支持千问和 OpenAI，向量库支持 pgvector 和 Milvus 随时切换，不改业务代码；再往上是领域层，这是我们投入最大的地方，八个业务域加上一个 Agent 引擎，全部用 Google Wire 在编译时注入，零反射、全部类型安全；最顶层是前端，React 加 Zustand，用 Ant Design 做 UI，核心交互是通过 SSE 接收 Agent 的流式事件实时渲染卡片。技术选型上，后端选 Go 是因为它的并发模型天然适合处理大量 SSE 长连接，而且编译成单个二进制部署极其方便；LLM 选 DeepSeek 是因为它在中文推理上性价比最优；前端选 Zustand 做状态管理，比 Redux 轻量很多，适合高频流式更新的场景。

数据库这里我快速过一下。左边是核心业务的 ER 图，七个实体，从部门到员工到报销单再到审批记录，最关键的亮点在票据表——我们设计了 OCR 审计链，有 OCR 的原始值、用户手动修正的值，以及审批人最终裁决选择哪一个，这样每一步都有据可查。金额全部用分为单位存储在数据库里，避免浮点精度问题，API 层再转成元。预算表采用三态模型，年度总额减去已结算、再减去冻结中的金额，才是真正可用的。右边是知识库和会话的 ER 图，政策文档被切成五百字符的块做向量嵌入，会话消息通过 Blades JSON 做完整序列化存储，支持任意轮次的上下文恢复。

AGENT DESIGN

Agent 两大设计模式

🧩 装饰器模式

RoleGuard — 运行时校验角色，fail-closed
Interruptable — 首次调用暂停，用户确认后执行
应用于 4 个敏感工具：提交 · 审批 · 取消
任意工具 + 装饰器 = 鉴权 + 确认 — 零代码改动

🤖 子 Agent 嵌套

独立 ReAct 循环 — 子Agent 自己思考→检索→判定
主Agent 无感知 — 只看干净 JSON 判断结果
同时有规则引擎（关键词+阈值）双路径保障
主Agent 只需关心结果 — 关注点分离 · Agent 可组合
设计哲学：关注点分离 · 零侵入扩展 · Agent 组合复用 · 人机协同确认
这一页是今天最核心的一页。我们 Agent 有两大设计模式。第一个是装饰器模式——我们做了两个零侵入的包装器，一个是 Interruptable，一个是 RoleGuard。任何工具只要在外面包一层 Interruptable，就自动获得了暂停等待用户确认的能力；包一层 RoleGuard，就自动获得了角色鉴权。你可以随时给新工具加上确认或鉴权，完全不改工具本身的代码。第二个是子 Agent 嵌套模式。合规审核不是一个简单的 if-else，而是一个独立的子 Agent。它内部有自己的 ReAct 推理循环，会先去向量库里检索相关政策文档，然后把政策文本和票据信息一起分析，最终给主 Agent 返回一个结构化的 JSON 结果。主 Agent 完全不需要知道子 Agent 内部是怎么推理的，实现了真正的关注点分离。这两种模式结合起来，让我们的 Agent 既安全又可扩展。

⚠️ 提交确认：唯一需人工介入 · 其余 7 步全部 Agent 自主完成（替代率 87.5%）

这里是员工报销的完整流程，一共四个阶段八步。第一阶段是信息采集，员工上传发票图片，Agent 自动调用多模态大模型做 OCR 识别，提取出发票上的金额、日期、类别、销售方这些信息。第二阶段是智能审核，Agent 先把识别出来的票据汇总展示给用户确认，然后自动去做合规审核——它会去向量库里检索公司报销政策，对每张票据做合规判断，同时检查部门预算是否充足。第三阶段是提交，合规和预算都通过之后，Agent 创建报销单草稿，这时候唯一需要人工介入的就是点击确认提交，触发了我们后面要讲的中断机制。用户确认后，系统生成审批链、冻结预算。第四阶段是自动归档，审批通过后系统生成 PDF 报销单、发送邮件通知。八步里面有七步是 Agent 自主完成的，人工替代率接近百分之八十八。

INTERRUPT

两阶段中断确认机制

❌ 无确认机制

LLM 直接调用 → 理解偏差 → 误操作 → 不可逆损失

✅ 两阶段确认

暂停 → 用户确认 → 重放工具 → 继续执行 · 单次有效防重放

关键状态字段
ApprovalState { Approved, Reason, Consumed }
session.State["pending_tool"]
session.State["approval"]
中断确认是我们 Agent 最核心的安全机制。当 LLM 决定要执行提交报销或者审批通过这种不可逆操作的时候，我们的 Interruptable 包装器会先把 Agent 暂停住，而不是直接执行。具体流程分两步：第一步，Agent 调用 submit_reimbursement，Interruptable 检测到这是首次调用，就把这个工具的信息存到会话状态里，然后给前端推送一个 interrupted 事件，Agent 暂停，前端弹出一个确认框。第二步，用户点了确认之后，浏览器发一个 POST 请求到 approve 接口，后端从会话状态里把这个工具拿出来重新执行一遍，这次 Interruptable 发现 Approved 是 true 而且没有被消费过，就真正去执行业务逻辑，同时把 Consumed 标记为 true。这个 Consumed 字段的妙处在于它是单次消费的——如果你试图重复提交同一个 approve 请求，第二次就会因为 Consumed 已经是 true 而被拒绝。这样就防止了重放攻击。对比一下，如果没有这个机制，LLM 理解偏差就直接误操作了，有了这个机制就是人机协同，既高效又安全。

前端这里我们做了一套很有意思的卡片渲染系统。后端通过 SSE 推送八种事件类型，前端把这些事件映射成四种可视化卡片。Thinking 和 Reasoning 事件合并成一张思考卡片，默认折叠的，用户点开可以看到 LLM 的推理过程，思考完成之后会有个动画效果。Tool Call 和 Tool Result 映射成工具卡片，每个工具独立一张，执行中用黄色标记，完成后变绿色——特别说一下 OCR 工具卡片里面还内嵌了一个可编辑的修正表单，用户可以手动修改识别结果。Message 事件是 LLM 的文字输出，用流式增量更新，支持 Markdown 表格渲染。Interrupted 事件触发一个黄色警告栏，用户点确认或者取消。整个过程通过 Zustand 的三个原子操作 appendCard、updateLastCard 和 finish 来控制，同一个类型连续来就追加到同一张卡片，类型切换就弹新卡片。这样 LLM 的推理过程对用户是完全透明的，而不是一个黑盒。

🧩
装饰器模式

7事件→4卡片 · 推理可视化 · 类型切换自动弹新

DB int64(分) → API float64(元) · 消除浮点精度 · 全链路转换

🔍
三层审计链

OCR原始值 → 用户修正 → 审批人裁决 · 完整追责

⚛️
向量+关键词双降级

Embedding不可用→关键词匹配 · 系统零中断

最后总结一下我们的八个核心创新点，分成两大类。智能体设计上有四个：装饰器模式让任何工具都可以零侵入地加上确认和鉴权；子 Agent 嵌套让合规审核成为一个独立的推理单元；两阶段中断用 ApprovalState 的一次性消费保证了安全；SSE 卡片渲染把 Agent 的黑盒推理变成了可视化。工程和数据处理上也有四个：角色感知 Agent 通过 Resolver 动态分配工具集；金额分层消除精度问题——数据库存分、API 传元、Agent 工具层自动转换；三层审计链让每一张票据从 OCR 原始值到最后审批人的裁决都有据可查；向量加关键词双降级保证了合规服务在任何情况下都不会中断。
