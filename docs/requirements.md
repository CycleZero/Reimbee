## 一、业务背景

### 1.1 问题陈述

企业日常费用报销是高频、刚需的业务场景。以一所高校的学院为例，每位教职工每年平均产生 15-30 笔报销申请，每笔报销涉及以下环节：

1. **票据收集**：员工收集发票、行程单、住宿水单等纸质或电子票据
2. **填单**：手工填写报销单，逐张录入发票信息（代码、号码、金额、日期）
3. **贴票**：按规定将票据粘贴在报销单背面
4. **签字审批**：逐级找领导签字（至少 2-3 级）
5. **财务审核**：财务处核对金额、检查合规性
6. **入账**：审核通过后录入财务系统

**当前流程的核心痛点**：

| 痛点 | 描述 | 影响 |
|------|------|------|
| **填单耗时** | 每张发票需手工录入 6-8 个字段，易出错 | 平均每笔报销填单耗时 10-15 分钟 |
| **合规靠人** | 差旅标准、招待限额依赖审批人记忆，无系统校验 | 超标报销经常到财务处才被发现，退回重走流程 |
| **预算盲区** | 员工/审批人看不到部门预算余额 | 年底集中超预算，财务被动 |
| **进度不透明** | 员工不知道报销流转到哪个节点，反复催问 | 沟通成本高，体验差 |
| **票据管理乱** | 纸质票据易丢失，电子发票散落在邮件/微信 | 归档困难，审计追溯成本高 |

### 1.2 业务目标

| 目标 | 衡量指标 | 期望值 |
|------|---------|:--:|
| 缩短报销填单时间 | 单笔报销从发起到提交的耗时 | 从 15 分钟降至 3 分钟以内 |
| 减少合规退回率 | 因不合规被财务退回的比例 | 从 30% 降至 5% 以下 |
| 提升预算可见性 | 员工提交前能查看部门预算余额的比例 | 100% |
| 改善进度透明度 | 员工能实时查看审批进度的比例 | 100% |
| 票据电子化 | 票据以电子形式归档的比例 | 100% |

### 1.3 业务范围

**在范围内**：

- 员工发起报销申请（上传票据、确认信息、提交）
- 系统自动识别票据信息（OCR）
- 系统自动校验合规性和预算
- 自动生成标准化报销单
- 通知审批人
- 员工查询报销进度和历史
- 部门预算使用情况可视化

**不在范围内**：

- 对接真实财务系统（ERP/OA）
- 实际资金支付/打款
- 发票真伪联网验证（税务系统对接）
- 移动端 App
- 多语言支持
- 与银行/税务系统的对接

## 二、干系人分析

### 2.1 干系人识别

| 干系人 | 角色描述 | 核心关注点 | 参与程度 |
|--------|---------|-----------|:--:|
| **普通员工** | 报销发起人，各部门教职工 | 填单快、不用记规则、随时知道进度 | 高频使用 |
| **部门审批人** | 部门主管/项目负责人，审批本部门报销 | 快速了解报销内容、预算是否充足、合规是否有问题 | 中频使用 |
| **财务人员** | 财务处审核人员，最终把关 | 票据合规、金额准确、预算可控 | 中频使用 |
| **系统管理员** | IT 运维人员 | 系统稳定、数据安全、可维护 | 低频使用 |

### 2.2 用户画像

#### 画像 A：普通员工（主要用户）

- **典型人物**：王老师，35 岁，计算机学院副教授
- **技术熟练度**：日常使用电脑和手机，但不喜欢复杂系统
- **报销频率**：每月 1-2 次，每次 2-5 张票据
- **痛点**："每次报销都要翻出差旅标准文件看住宿限额，填完单还要跑三层楼找领导签字，不知道什么时候能到账"
- **期望**："拍个照就能报销，系统告诉我符不符合规定，不用我自己算"

#### 画像 B：部门审批人

- **典型人物**：张主任，48 岁，学院副院长
- **技术熟练度**：基本电脑操作，偏好简洁界面
- **审批频率**：每周 5-10 笔
- **痛点**："报销单上信息太多，我只关心总金额和有没有超标。有时候批完了才发现部门预算快用完了"
- **期望**："收到通知点开就能看到关键信息（谁、多少钱、什么类别、预算还够不够），一键通过"

#### 画像 C：财务人员

- **典型人物**：李会计，30 岁，财务处审核岗
- **技术熟练度**：熟练使用各类办公和财务软件
- **审核频率**：每天 10-20 笔
- **痛点**："发票信息经常录错，超标问题到我这才发现，退回去重新走流程，大家都麻烦"
- **期望**："到我这的单子应该是已经校验过的，我只需做最终确认，不需要逐张核对发票"

## 三、业务流程分析

### 3.1 当前业务流程（AS-IS）

```
员工                      审批人                    财务
  │                         │                        │
  ├─ 收集票据                │                        │
  ├─ 手工填报销单             │                        │
  ├─ 粘贴票据                 │                        │
  ├─ 找领导签字 ────────────▶ ├─ 审阅内容               │
  │                         ├─ 签字 ─────────────────▶ ├─ 核对发票
  │                         │                        ├─ 校验合规（人工）
  │                         │                        ├─ 检查预算（人工）
  │  ◀────────── 退回重填 ───┼──────── 退回 ──────────┤
  │                         │                        ├─ 录入财务系统
  │                         │                        └─ 归档
  │                         │
  ├─ 反复催问进度             │
  └─ 等待到账                 │
```

**关键问题**：
1. 合规校验在最后一个环节（财务处）才做，退回成本高
2. 预算信息不对员工和审批人可见
3. 进度完全靠人工询问，无系统支持

### 3.2 目标业务流程（TO-BE）

```
员工                              审批人                    财务
  │                                 │                        │
  ├─ 拍照上传票据 ──────────────────┤                        │
  ├─ 系统自动 OCR 识别 ◀────────────┤                        │
  ├─ 系统自动合规校验 ◀──────────────┤                        │
  ├─ 系统自动预算校验 ◀──────────────┤                        │
  ├─ 确认提交 ──────────────────────┤                        │
  │                                 │                        │
  │              系统自动生成 PDF ───┼── 邮件推送 ────────────▶ │
  │                                 ├─ 收到通知，一键审批 ────▶ ├─ 收到已校验的报销单
  │  ◀── 实时收到审批进度更新 ──────┤                        ├─ 最终确认
  │                                 │                        └─ 归档
  │                                 │
  └─ 随时查看预算看板                │
```

**改进要点**：
1. 合规校验前置：提交时就自动完成，问题在源头解决
2. 预算前置可见：提交前员工和审批人可看到预算余额
3. 进度透明：每个节点变更自动通知
4. 票据电子化：上传即归档，无需纸质流转

### 4.2 F1：智能对话报销（核心功能）

#### 4.2.1 功能概述

用户通过与智能助手进行自然语言对话，完成从发票上传到报销提交的全流程。助手理解用户意图，引导用户完成必要步骤。

#### 4.2.2 用户故事

| ID | 作为... | 我想要... | 以便... | 优先级 |
|:--:|--------|----------|--------|:--:|
| US01 | 普通员工 | 通过对话方式发起报销 | 不用学习复杂系统操作 | P0 |
| US02 | 普通员工 | 上传发票图片后自动识别金额和类别 | 不用手动录入发票信息 | P0 |
| US03 | 普通员工 | 提交前知道报销是否符合公司规定 | 避免被退回重走流程 | P0 |
| US04 | 普通员工 | 提交前知道部门预算是否充足 | 避免超预算被拒 | P0 |
| US05 | 普通员工 | 提交后自动通知审批人 | 不用自己去催审批 | P0 |
| US06 | 普通员工 | 上传多张发票时自动汇总金额 | 不用自己计算 | P1 |
| US07 | 普通员工 | 发现识别错误时可以手动修正 | 确保报销信息准确 | P1 |

### 4.3 F2：预算看板

#### 4.3.1 功能概述

以可视化图表展示各部门的年度预算使用情况，帮助员工和审批人在提交/审批报销前了解预算余量。

#### 4.3.2 用户故事

| ID | 作为... | 我想要... | 以便... | 优先级 |
|:--:|--------|----------|--------|:--:|
| US10 | 普通员工 | 提交报销前看到部门预算余额 | 判断是否需要特殊审批 | P1 |
| US11 | 审批人 | 审批时看到部门预算使用率 | 快速判断是否批准 | P1 |
| US12 | 财务人员 | 查看所有部门的预算使用情况 | 及时预警超支风险 | P1 |

#### 4.3.3 展示内容

**概览区**：
- 全公司预算总额、已使用金额、剩余金额
- 总体预算使用率（百分比 + 进度条）

**详细区（按部门）**：
- 部门名称
- 年度预算总额
- 已使用金额
- 待审批金额（已提交但未审批的冻结金额）
- 可用余额（= 年度预算 - 已使用 - 待审批）
- 使用率（百分比）
- 状态标识（正常 / 预警 / 超支）

**可视化要求**：
- 各部门预算使用率对比（图表形式）
- 各部门已使用 vs 待审批金额对比（图表形式）
- 使用率超过 80% 的部门突出标识

#### 4.3.4 业务规则

| 规则编号 | 条件 | 状态 | 颜色 |
|:--:|------|:--:|:--:|
| BR10 | 使用率 < 60% | 正常 | 绿色 |
| BR11 | 60% ≤ 使用率 < 80% | 关注 | 黄色 |
| BR12 | 80% ≤ 使用率 < 95% | 预警 | 橙色 |
| BR13 | 使用率 ≥ 95% | 严重 | 红色 |

### 4.4 F3：报销进度查询

#### 4.4.1 功能概述

员工可随时查询自己提交的报销单当前审批进度，了解流转到哪个审批节点。

#### 4.4.2 用户故事

| ID | 作为... | 我想要... | 以便... | 优先级 |
|:--:|--------|----------|--------|:--:|
| US20 | 普通员工 | 输入报销单号查询审批状态 | 不用催审批人就能知道进度 | P1 |
| US21 | 普通员工 | 按日期查看自己的所有报销记录 | 了解历史报销情况 | P1 |
| US22 | 普通员工 | 看到每个审批节点的审批人、状态、意见 | 了解驳回原因或补充材料要求 | P1 |

#### 4.4.3 进度展示内容

**单笔报销详情**：
- 报销单号、提交时间、总金额
- 审批链（按顺序展示每个节点的状态和时间线）：
  - 节点名称（如"部门主管审批"）
  - 审批人姓名
  - 状态（待审批 / 已通过 / 已驳回）
  - 审批意见
  - 操作时间

**状态视觉映射**：
- 待审批：灰色 + 等待图标
- 已通过：绿色 + 勾选图标
- 已驳回：红色 + 叉号图标
- 当前节点：蓝色 + 加载动画

#### 4.4.4 审批链定义

一个报销单提交后，按以下顺序流转（固定的审批链）：

| 审批节点 | 触发条件 | 说明 |
|---------|---------|------|
| 部门主管审批 | 总是触发 | 员工所在部门的主管 |
| 财务复核 | 总是触发 | 财务处的审核人员 |
| 总监审批 | 报销金额 > 阈值（默认 ¥10,000） | 大额报销需额外审批 |
| 特殊审批 | 预算超支触发 | 部门预算不足时的额外审批环节 |

### 4.5 F4：后台管理

#### 4.5.1 功能概述

提供审批操作入口和基础数据管理能力（当前阶段以模拟方式实现）。

#### 4.5.2 用户故事

#### 4.5.3 模拟审批说明

项目演示阶段，审批流程通过模拟方式实现：报销单提交后，系统自动模拟审批流程——每个审批节点间隔一定时间自动通过（或按预设比例随机通过/驳回），以展示完整的业务流转效果。

### 5.1 费用类别规则

| 类别 | 涵盖范围 | 示例票据 |
|------|---------|---------|
| 差旅-交通 | 飞机票、火车票、长途汽车票、市内交通 | 机票行程单、高铁票、打车发票 |

### 5.2 合规校验规则

| 规则编号 | 费用类别 | 校验项 | 阈值 | 超阈值处理 |
|:--:|---------|--------|:--:|------|
| BR01 | 差旅-住宿 | 单日住宿费 | ≤ ¥300/天 | 超出部分标记 warning，需审批人确认 |
| BR02 | 差旅-交通（机票） | 单次机票 | ≤ ¥1,500（经济舱全价） | 超出标记 error，不可提交 |
| BR03 | 差旅-交通（火车） | 单次火车票 | 一等座/软卧需说明理由 | 超出标记 warning |
| BR04 | 招待费 | 人均消费 | ≤ ¥200/人 | 超出标记 error，需提供招待对象和事由 |
| BR05 | 办公用品 | 单次总额 | ≤ ¥5,000 | 超出标记 warning，需提供明细清单 |
| BR06 | 所有类别 | 发票开具日期 | 距提交日期 ≤ 90 天 | 超出标记 error，不可提交 |

**校验结果等级**：

| 等级 | 含义 | 处理方式 |
|:--:|------|---------|
| pass | 完全合规 | 正常提交 |
| warning | 存在关注项，但不阻止提交 | 提示用户，用户确认后可提交；审批人可见 warning 标记 |
| error | 严重违规 | 阻止提交，引导用户修正 |

### 5.3 预算控制规则

| 规则编号 | 场景 | 规则 | 处理 |
|:--:|------|------|------|
| BR10 | 提交报销 | 本次金额 ≤ 部门可用余额 | 正常提交 |
| BR11 | 提交报销 | 本次金额 > 部门可用余额，但 ≤ 总预算的 110% | 触发特殊审批流程 |
| BR12 | 提交报销 | 本次金额 > 总预算的 110% | 禁止提交，需联系财务调整预算 |
| BR13 | 审批通过 | 冻结金额释放为实际支出 | 更新 spent_amount，释放 frozen_amount |
| BR14 | 审批驳回 | 冻结金额解冻 | 释放 frozen_amount，恢复可用余额 |

### 5.4 报销单编号规则

- 格式: `REIMB-{年份}-{4 位流水号}`
- 示例: `REIMB-2026-0001`
- 流水号按年度重置，从 0001 开始递增

### 7.1 易用性

| 要求 | 描述 | 优先级 |
|------|------|:--:|
| 对话式交互 | 用户通过自然语言与系统交互，无需学习复杂操作流程 | P0 |

| 引导式操作 | 系统主动引导用户完成必要步骤，而非让用户自己探索 | P0 |

| 多轮对话连贯 | 对话中断后恢复时，保留之前的上下文（至少最近 3 轮） | P1 |

| 场景 | 期望行为 | 优先级 |
|------|---------|:--:|

| 多用户同时使用 | 不同用户的对话互不干扰 | P1 |
| 系统异常崩溃 | 正在进行的对话可恢复 | P2 |

### 7.3 安全性

| 要求 | 描述 | 优先级 |
